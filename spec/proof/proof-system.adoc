== Abstract

This document specifies the proof system of the TypeLL verification kernel.
When the type checker encounters constraints it cannot automatically decide
(refinement predicates with variables, dependent type index equalities over
symbolic terms, session protocol completeness), it generates _proof
obligations_ — logical propositions that must be proved true for the program
to be well-typed.

TypeLL operates as a _proof obligation generator_, not a theorem prover.
Generated obligations are either discharged by a lightweight built-in
decision procedure or forwarded to external provers (ECHIDNA, SMT solvers).
Successfully proved obligations produce _proof certificates_ that can be
attached to compiled artefacts as _proof-carrying code_.

The specification maps directly to the Rust implementation in
`crates/typell-core/src/proof.rs`.

== Proof Obligation Structure

=== Data Type

A proof obligation is a record with the following fields:

[cols="2,1,4"]
|===
| Field | Type | Description

| `id`
| `String`
| Unique identifier of the form `PO-NNNN` (zero-padded 4-digit sequence).
  Stable within a single checking session.

| `description`
| `String`
| Human-readable summary (e.g., `"Refinement: x > 0 on Int"`).

| `proposition`
| `Proposition`
| The logical proposition to prove. See <<propositions>>.

| `origin`
| `ObligationOrigin`
| Where in the type-checking process this obligation was generated.
  See <<origins>>.

| `status`
| `ObligationStatus`
| Current discharge status. See <<statuses>>.
|===

Rust mapping: `proof.rs::ProofObligation`.

[source,rust]
----
pub struct ProofObligation {
    pub id: String,
    pub description: String,
    pub proposition: Proposition,
    pub origin: ObligationOrigin,
    pub status: ObligationStatus,
}
----

[[propositions]]
=== Proposition Variants

A `Proposition` is the logical statement that must be proved:

[cols="2,4"]
|===
| Variant | Description

| `PredicateHolds(Predicate)`
| A refinement predicate must be provably true. Generated when a value is
  used at a refined type and the predicate cannot be statically evaluated.

| `TermEquality(Term, Term)`
| Two type-level terms must be provably equal. Generated during dependent
  type unification when symbolic terms cannot be reduced to literals.

| `WellFounded(Term)`
| A term must be well-founded, i.e., the recursion defined by this term
  terminates. Generated for recursive definitions.

| `DimensionCompatible(String, String)`
| Two dimensional annotations must be compatible for a given operation.
  Generated from `Resource` type arithmetic.

| `SessionComplete(String)`
| A session type protocol must reach `End` (no dangling sends or receives).
  Generated when the type checker cannot statically verify protocol
  completeness.

| `Custom(String)`
| Escape hatch for domain-specific propositions from language bridges.
|===

Rust mapping: `proof.rs::Proposition`.

[[origins]]
=== Obligation Origins

The `ObligationOrigin` records the source of the obligation:

[cols="2,2,4"]
|===
| Origin | Fields | Generated By

| `Refinement`
| `base_type: String`, `predicate: String`
| The refinement type checker, when a predicate on a refined type cannot
  be statically evaluated.

| `DependentIndex`
| `type_expr: String`
| The unification engine, when two dependent type indices contain variables
  and cannot be reduced.

| `UserAnnotation`
| `annotation: String`
| Explicit `assert` or `prove` annotations in user code.

| `SessionProtocol`
| `protocol: String`
| The session type checker, when protocol completeness cannot be statically
  determined.

| `DimensionalAnalysis`
| `operation: String`
| The dimensional type checker, when dimension compatibility cannot be
  determined at compile time.
|===

Rust mapping: `proof.rs::ObligationOrigin`.

[[statuses]]
=== Obligation Statuses

The lifecycle of a proof obligation:

[cols="2,4"]
|===
| Status | Description

| `Pending`
| The obligation has been generated but not yet attempted. It awaits
  automatic discharge or forwarding to an external prover.

| `Discharged`
| The obligation was automatically discharged by TypeLL's built-in
  decision procedure.

| `ProvedExternally(String)`
| The obligation was discharged by an external prover. The string
  identifies the prover (e.g., `"ECHIDNA"`, `"Z3"`, `"CVC5"`).

| `Unresolved`
| The obligation could not be proved by any available prover. The user
  must provide a manual proof or restructure the code.

| `Refuted(String)`
| The proposition was proved _false_. The program is ill-typed. The
  string contains the refutation reason.
|===

Rust mapping: `proof.rs::ObligationStatus`.

== Proof Obligation Generation

=== The Obligation Collector

Proof obligations are accumulated during type checking by the
`ObligationCollector`:

[source,rust]
----
pub struct ObligationCollector {
    obligations: Vec<ProofObligation>,
    next_id: u32,
}
----

Each obligation receives a unique ID via `fresh_id()`:

[source,rust]
----
fn fresh_id(&mut self) -> String {
    let id = format!("PO-{:04}", self.next_id);
    self.next_id += 1;
    id
}
----

=== Generation from Refinement Types

When the type checker encounters a refined type stem:[\{x : A \mid P(x)\}]
and the predicate stem:[P(x)] contains variables:

[stem]
++++
\frac{
  \Gamma \vdash e : A
  \quad
  \Gamma \vdash e : \{x : A \mid P(x)\}
  \quad
  \mathrm{eval}(P(x)) = \mathsf{Unknown}
}{
  \text{generate } \mathsf{PO}: \mathsf{PredicateHolds}(P(x))
} \quad (\text{GEN-REFINE})
++++

Rust mapping: `ObligationCollector::add_refinement()`.

[source,rust]
----
pub fn add_refinement(
    &mut self,
    predicate: Predicate,
    base_type: &str,
) -> &ProofObligation {
    let id = self.fresh_id();
    let desc = format!("Refinement: {:?} on {}", predicate, base_type);
    self.obligations.push(ProofObligation {
        id,
        description: desc,
        proposition: Proposition::PredicateHolds(predicate.clone()),
        origin: ObligationOrigin::Refinement {
            base_type: base_type.to_string(),
            predicate: format!("{:?}", predicate),
        },
        status: ObligationStatus::Pending,
    });
    self.obligations.last().unwrap()
}
----

=== Generation from Dependent Type Indices

When unification encounters two type-level terms that cannot be reduced:

[stem]
++++
\frac{
  \Gamma \vdash t_1 : \mathsf{Term}
  \quad
  \Gamma \vdash t_2 : \mathsf{Term}
  \quad
  \mathrm{eval}(t_1) \ne \mathrm{eval}(t_2)
  \quad
  t_1, t_2 \text{ contain variables}
}{
  \text{generate } \mathsf{PO}: \mathsf{TermEquality}(t_1, t_2)
} \quad (\text{GEN-TERM-EQ})
++++

Rust mapping: `ObligationCollector::add_term_equality()`.

=== Generation from Session Types

When a session type endpoint goes out of scope or when branching makes
completeness uncertain:

[stem]
++++
\frac{
  c :^{1} \mathsf{Chan}\langle S \rangle \in \Gamma
  \quad
  S \ne \mathsf{End}
  \quad
  \text{scope exit}
}{
  \text{generate } \mathsf{PO}: \mathsf{SessionComplete}(\mathrm{name}(S))
} \quad (\text{GEN-SESSION})
++++

Rust mapping: `ObligationCollector::add_session_completeness()`.

== Built-In Decision Procedure

=== Overview

TypeLL includes a lightweight decision procedure that attempts to
discharge obligations without external provers. This handles the common
case of constant predicates and syntactic term equalities.

=== Predicate Evaluation

The predicate evaluator (`eval_predicate`) handles:

==== Constant Folding

If both operands of a comparison are literals, the comparison is evaluated
directly:

[stem]
++++
\frac{
  \mathrm{eval}(t_1) = n_1 \quad \mathrm{eval}(t_2) = n_2
}{
  \mathrm{eval}(t_1 > t_2) = (n_1 > n_2)
} \quad (\text{PRED-CONST})
++++

Example: `5 > 0` evaluates to `PredicateResult::True`.

==== Boolean Combinations

Short-circuit evaluation for conjunctions and disjunctions:

[stem]
++++
\mathrm{eval}(P_1 \wedge P_2) = \begin{cases}
  \mathsf{False} & \text{if } \mathrm{eval}(P_1) = \mathsf{False} \vee \mathrm{eval}(P_2) = \mathsf{False} \\
  \mathsf{True}  & \text{if } \mathrm{eval}(P_1) = \mathsf{True} \wedge \mathrm{eval}(P_2) = \mathsf{True} \\
  \mathsf{Unknown} & \text{otherwise}
\end{cases}
++++

[stem]
++++
\mathrm{eval}(P_1 \vee P_2) = \begin{cases}
  \mathsf{True}  & \text{if } \mathrm{eval}(P_1) = \mathsf{True} \vee \mathrm{eval}(P_2) = \mathsf{True} \\
  \mathsf{False} & \text{if } \mathrm{eval}(P_1) = \mathsf{False} \wedge \mathrm{eval}(P_2) = \mathsf{False} \\
  \mathsf{Unknown} & \text{otherwise}
\end{cases}
++++

[stem]
++++
\mathrm{eval}(\neg P) = \begin{cases}
  \mathsf{False} & \text{if } \mathrm{eval}(P) = \mathsf{True} \\
  \mathsf{True}  & \text{if } \mathrm{eval}(P) = \mathsf{False} \\
  \mathsf{Unknown} & \text{otherwise}
\end{cases}
++++

==== Contradiction Detection

If any conjunct is provably false, the entire conjunction is false:

[source]
----
-- x > 5 AND 3 > 10 — second conjunct is false
-- Therefore the entire predicate is false
eval(And(Gt(Var("x"), Lit(5)), Gt(Lit(3), Lit(10)))) = False
----

==== Raw Predicates

`Predicate::Raw` (opaque strings) always evaluate to `Unknown`.

=== Term Equality

The term equality check uses syntactic equality:

[stem]
++++
\mathrm{terms\_equal}(t_1, t_2) = (t_1 \equiv t_2)
\quad (\text{TERM-EQ-SYN})
++++

where stem:[\equiv] denotes structural equality (Rust's `PartialEq`).

NOTE: Semantic equality (e.g., recognising that `a + b = b + a`) requires
an external prover and is not handled by the built-in procedure.

=== Automatic Discharge Pipeline

The `try_discharge_refinements()` method iterates over all pending
obligations and attempts to discharge them:

[source,rust]
----
pub fn try_discharge_refinements(&mut self) -> usize {
    let mut discharged = 0;
    for obligation in &mut self.obligations {
        if obligation.status != ObligationStatus::Pending {
            continue;
        }
        if let Proposition::PredicateHolds(ref pred) = obligation.proposition {
            match eval_predicate(pred) {
                PredicateResult::True => {
                    obligation.status = ObligationStatus::Discharged;
                    discharged += 1;
                }
                PredicateResult::False => {
                    obligation.status = ObligationStatus::Refuted(
                        "predicate is statically false".to_string(),
                    );
                    discharged += 1;
                }
                PredicateResult::Unknown => { /* remains pending */ }
            }
        }
        if let Proposition::TermEquality(ref t1, ref t2) = obligation.proposition {
            if terms_equal(t1, t2) {
                obligation.status = ObligationStatus::Discharged;
                discharged += 1;
            }
        }
    }
    discharged
}
----

== SMT Integration with ECHIDNA

=== Architecture

TypeLL generates proof obligations; ECHIDNA discharges them. The interaction
follows a request-response pattern:

. TypeLL type-checks a program and collects obligations.
. TypeLL's built-in decision procedure handles trivial obligations.
. Remaining `Pending` obligations are serialised as JSON and included in
  the `CheckResult.proof_obligations` array.
. The PanLL client forwards pending obligations to ECHIDNA via its API.
. ECHIDNA translates obligations to SMT-LIB format and invokes a solver.
. ECHIDNA returns discharge results (proved, refuted, or timeout).
. The client updates obligation statuses and reports to the user.

=== Obligation-to-SMT Translation

[cols="2,4"]
|===
| Proposition | SMT-LIB Translation

| `PredicateHolds(Gt(Var("x"), Lit(0)))`
| `(assert (> x 0))` / `(check-sat)`

| `TermEquality(BinOp(Add, Var("n"), Var("m")), BinOp(Add, Var("m"), Var("n")))`
| `(assert (not (= (+ n m) (+ m n))))` / `(check-sat)` (refute to prove)

| `WellFounded(App("f", [BinOp(Sub, Var("n"), Lit(1))]))`
| Termination checking via ranking function synthesis

| `DimensionCompatible("kg*m/s^2", "N")`
| Dimensional algebra (handled internally, not SMT)

| `SessionComplete("BuyProtocol")`
| Model checking via protocol state space exploration
|===

NOTE: Full ECHIDNA integration is planned for Phase 2.

== Proof Certificates

=== Structure

A proof certificate records the successful discharge of a proof obligation:

[cols="2,1,4"]
|===
| Field | Type | Description

| `obligation_id`
| `String`
| The `PO-NNNN` identifier of the discharged obligation.

| `hash`
| `String`
| SHA-256 hash of the obligation's proposition, ensuring the certificate
  matches the obligation.

| `timestamp`
| `String`
| ISO 8601 timestamp of when the proof was completed.

| `verifier`
| `String`
| Identifier of the entity that discharged the obligation:
  `"typell-builtin"`, `"echidna"`, `"z3"`, `"cvc5"`, or a user identifier.

| `proof_term`
| `Option<String>`
| Optional proof term (lambda-calculus witness) for constructive proofs.
|===

=== Certificate Verification

A proof certificate is _valid_ if:

. The `hash` matches `SHA256(canonical_serialization(obligation.proposition))`.
. The `verifier` is in the trusted verifier list.
. The `timestamp` is within the certificate validity window (configurable;
  default: no expiry).

== Proof-Carrying Code

=== ProvedResult<T>

TypeLL supports _proof-carrying code_ through the `ProvedResult<T>` type.
A `ProvedResult<T>` is a value of type `T` bundled with proof certificates
for all obligations generated during its type checking:

[source]
----
type ProvedResult<T> = {
    value: T,
    certificates: Vec<ProofCertificate>,
    obligations_total: u32,
    obligations_discharged: u32,
    obligations_pending: u32,
}
----

=== Proof Coverage

A `ProvedResult<T>` has _full proof coverage_ when all obligations are
discharged:

[stem]
++++
\frac{
  \mathsf{obligations\_pending} = 0
  \quad
  \forall c \in \mathsf{certificates}.\; \mathsf{valid}(c)
}{
  \mathsf{fully\_proved}(\mathsf{ProvedResult}\langle T \rangle)
} \quad (\text{FULL-PROOF})
++++

=== Consumption

A `ProvedResult<T>` can be consumed in two ways:

. **Trusted extraction**: If fully proved, extract the value with no
  runtime cost.
. **Checked extraction**: If partially proved, extract the value but emit
  warnings for unresolved obligations.

== Proof Composition

=== Transitivity

If stem:[P \Rightarrow Q] and stem:[Q \Rightarrow R] are both proved, then
stem:[P \Rightarrow R] is proved:

[stem]
++++
\frac{
  \mathsf{cert}_1 : P \Rightarrow Q
  \quad
  \mathsf{cert}_2 : Q \Rightarrow R
}{
  \mathsf{compose}(\mathsf{cert}_1, \mathsf{cert}_2) : P \Rightarrow R
} \quad (\text{PROOF-TRANS})
++++

=== Conjunction

If stem:[P] and stem:[Q] are both proved, then stem:[P \wedge Q] is proved:

[stem]
++++
\frac{
  \mathsf{cert}_1 : P
  \quad
  \mathsf{cert}_2 : Q
}{
  \mathsf{and}(\mathsf{cert}_1, \mathsf{cert}_2) : P \wedge Q
} \quad (\text{PROOF-CONJ})
++++

=== Weakening

A proved proposition remains proved when additional hypotheses are added:

[stem]
++++
\frac{
  \mathsf{cert} : P
}{
  \mathsf{weaken}(\mathsf{cert}, Q) : Q \Rightarrow P
} \quad (\text{PROOF-WEAK})
++++

=== Substitution

If stem:[P(x)] is proved for a specific value stem:[a], and we need
stem:[P(b)] where stem:[a = b] is also proved:

[stem]
++++
\frac{
  \mathsf{cert}_1 : P(a)
  \quad
  \mathsf{cert}_2 : a = b
}{
  \mathsf{subst}(\mathsf{cert}_1, \mathsf{cert}_2) : P(b)
} \quad (\text{PROOF-SUBST})
++++

== Counterexample Extraction

=== Overview

When an obligation is refuted (status `Refuted`), the proof system
extracts a _counterexample_ — a concrete assignment of values to variables
that makes the proposition false. This helps the programmer understand
why the program is ill-typed.

=== Extraction Process

. The built-in decision procedure identifies a `False` result.
. It constructs a counterexample from the literal values that caused the
  refutation.
. For external provers, the SMT solver's `(get-model)` command provides
  the counterexample.

=== Example

[source]
----
type PosInt = {x : Int | x > 0}

fn safeDivide(a : Int, b : PosInt) -> Int = a / b

-- Call with a variable:
let n : Int = readInt()
safeDivide(10, n)
  -- PO-0000: PredicateHolds(Gt(Var("n"), Lit(0)))
  -- Status: Pending (cannot determine statically)

-- Call with a negative literal:
safeDivide(10, -5)
  -- PO-0001: PredicateHolds(Gt(Lit(-5), Lit(0)))
  -- Status: Refuted("predicate is statically false")
  -- Counterexample: n = -5, which makes (n > 0) false
----

=== Counterexample for Term Equality

[source]
----
fn append<n, m>(xs : Vec<n, Int>, ys : Vec<m, Int>) -> Vec<n * m, Int>
  -- PO-0000: TermEquality(BinOp(Add, Var("n"), Var("m")),
  --                         BinOp(Mul, Var("n"), Var("m")))
  -- This is likely a bug: the programmer wrote n*m instead of n+m
  -- Counterexample: n=2, m=3 => 2+3=5 ≠ 2*3=6
----

== Obligation Lifecycle

=== Complete Flow

. **Generation**: Type checker creates obligation via `ObligationCollector`.
. **Auto-discharge**: `try_discharge_refinements()` attempts trivial cases.
. **Serialisation**: Remaining obligations appear in `CheckResult.proof_obligations`.
. **External proving**: PanLL forwards to ECHIDNA (Phase 2).
. **Certificate creation**: Proved obligations receive certificates.
. **Bundling**: Certificates are attached to `ProvedResult<T>`.
. **Audit**: Certificates are logged for compliance verification.

=== State Machine

[source]
----
        ┌──────────┐
        │ Pending  │
        └────┬─────┘
             │
    ┌────────┼────────┐
    ▼        ▼        ▼
┌────────┐ ┌──────┐ ┌─────────┐
│Dischargd│ │Refutd│ │Unresolvd│
└────────┘ └──────┘ └─────────┘
    │                    │
    ▼                    ▼
┌──────────────┐   (user action
│ProvedExternly│    required)
└──────────────┘
----
